# TASK-060 — Разоблачения как архив исследовательских дел с единым шаблоном

**Статус:** В очереди
**Приоритет:** Высокий
**Метки:** ui, exposure, research, architecture, data-migration

## Проблема (уточнено после аудита кода)

Сейчас под «Разоблачения» скрываются четыре несвязанные системы:

- `data/researches.json` + `js/load-researches.js` (route `#researches`) — реальный источник ленты в сайдбаре, плоский `body`-markdown без секций.
- `data/research/*.json` + инлайновый рендер в `index.html` (route `#research-library`) — более богатая схема (tags, related, content-секции), но отдельный route.
- `data/exposure.json` + `renderDocumentPage` — длинные эссе-документы без структуры.
- `js/exposure-editor.js` — форма авторинга, пишет только в `localStorage`, не связана ни с одним хранилищем выше.

Пока эти четыре системы существуют параллельно, единый шаблон невозможен — карточка, отрендеренная одним модулем, не сможет открыть дело, созданное в другом. Пункт 0 — обязательное условие для всего остального.

Реальный путь модуля: `products/website/apps/researchlab/` (путь `researchlab/` в CLAUDE.md устарел).

## Цель

Один JSON-шаблон дела. Один JS-рендерер, который берёт этот JSON и рисует и карточку в каталоге, и полноценную страницу-досье, и live-preview в форме создания. Автор создаёт дело в форме → видит точно то же, что увидит читатель.

## 0. Архитектурное решение (сделать первым, до UI)

- Ввести единую схему `data/exposures/<slug>.json` (один файл — одно дело, по аналогии с `data/research/*.json` — эта схема ближе всего к цели, берём её за основу и расширяем).
- Написать разовый скрипт миграции (`tools/migrate-exposures.py`) — конвертирует `researches.json` + `research/*.json` в новую схему. После проверки — старые файлы и их рендереры удаляются, а не остаются рядом.
- `data/exposure.json` (длинные эссе) — отдельный тип контента, не карточка-дело. Не мигрировать в ту же схему; оставить как документ или явно решить исключить из скоупа.

## 1. Единая схема дела

```json
{
  "id": "korban",
  "slug": "korban",
  "title": "Жертва → קרבן",
  "summary": "...",
  "tags": ["жертва", "юридизация"],
  "category": "языковые-подмены",
  "distortionType": "юридизация",
  "method": "этимологическая-хирургия",
  "period": "...",
  "status": "published",
  "confidence": "verified",
  "sources": [{ "type": "TM", "ref": "...", "url": "" }],
  "related": ["hesed", "teshuva"],
  "roots": ["קרב"],
  "updatedAt": "2026-07-19",
  "createdAt": "...",
  "author": "...",
  "changelog": [{ "date": "...", "note": "..." }],
  "sections": {
    "thesis": "...",
    "original": { "hebrew": "", "translit": "", "root": "", "paleo": "" },
    "shift": "...",
    "transmissionChain": [{ "layer": "иврит", "word": "" }, { "layer": "греческий", "word": "" }],
    "evidence": [{ "type": "Qumran", "ref": "", "note": "" }],
    "reconstruction": "...",
    "caveats": [{ "text": "...", "kind": "гипотеза" }],
    "relatedMaterials": []
  }
}
```

`confidence` — enum (`verified | needs-review | hypothesis | disputed`), не свободный текст как сейчас.
`status` — enum (`draft | review | published`), взят из `exposure-editor.js`, где он уже существует.

## 2. Единый рендерер

Новый модуль `js/modules/exposure-case.js` с тремя точками входа на одних и тех же функциях:

- `renderCard(data)` — для каталога.
- `renderCase(data)` — полное досье (breadcrumb, sticky TOC, читаемая колонка, sticky infobox, progress bar, badges доверия, evidence-таблица, визуальная цепочка передачи, CTA-кнопки).
- `renderPreview(data)` — то же, что `renderCase`, но встраивается внутрь формы создания как live-preview.

Старые рендереры удаляются: detail-путь в `load-researches.js`, `LabRenderer.renderResearchDetail` в `index.html`. `exposure-editor.js` переиспользуется как форма, но переписывается под новую схему и вызывает `renderPreview` вместо собственного превью.

## 3. Кнопка создания нового дела

На landing-экране `#exposure` — кнопка «+ Новое дело». Ведёт на `#exposure-editor` (существующий route, переиспользуется) — форма по полям схемы из п.1, с шагами (тезис → оригинал → сдвиг → цепочка → свидетельства → реконструкция → оговорки), автосохранением черновика в `localStorage` (как сейчас) и панелью live-preview на основе `renderPreview`.

**Вопрос персистентности.** Проект — статический сайт на GitHub Pages без бэкенда для записи данных (сервер на 8000 — только запуск AI-агентов через subprocess, не хранилище). Поэтому «создание дела» не может сразу попасть в общий каталог для всех читателей. Решение: кнопка «Экспортировать JSON» в форме генерирует файл по схеме из п.1 — автор коммитит его в `data/exposures/` вручную (как сейчас делает `tools/generators/generate-researches-json.py` из markdown). Публикация без git — отдельная задача (нужен endpoint на `server.py` для записи файла + валидация), в скоуп не входит.

## 4. Каталог, поиск, фильтры, доверие

Из исходной задачи: каталог карточек, полнотекстовый поиск (title, summary, заголовки секций, tags), фильтры (язык/слой, тип искажения, метод, сфера, уровень уверенности), сортировка, chips, debounce, empty/error/loading states, deep-link в hash, восстановление состояния — применяются к единой схеме и единому рендереру.

Бейджи доверия: Проверено / Требует проверки / Гипотеза / Спорно (на основе enum `confidence`). Раздел «Что мы знаем точно / Что предполагаем». Дата последней проверки и `changelog`.

## 5. Router

`LabRouter` (`js/router.js`) сейчас не парсит query-параметры — только конкатенация id в хэш вида `research-library-<id>`. Нужно расширить `handleHash()` для парсинга `#exposure/case/<slug>?tag=...` и сериализации фильтров каталога обратно в хэш при изменении.

## 6. Санитизация (P0, не P1)

`marked.parse()` льётся в `innerHTML` без очистки в 4 местах в текущем коде. С появлением формы создания источник markdown становится untrusted. Подключить DOMPurify один раз — в едином рендерере из п.2, единственной точке, через которую теперь проходит весь markdown.

## 7. AI-интеграция (понижен приоритет)

Кнопка «Попросить AI дополнить» в детальном просмотре — заглушка (добавляет пункт в `changelog` со статусом «запрошено»), не прямой fetch на `localhost:8000`. Сервер агентов (`products/agents/server.py`) сейчас без авторизации и с открытым CORS; реальная интеграция — отдельная задача после защиты эндпоинта.

## Приоритеты

**P0:** миграция данных (п.0-1), единый рендерер (п.2), каталог+поиск+фильтры на новой схеме, детальный просмотр, санитизация.

**P1:** форма создания с live-preview и экспортом (п.3), deep-link роутинг (п.5), связи с `roots.json`.

**P2:** реальная AI-интеграция (после защиты сервера), граф связей «слово → корень → перевод → метод → стих», side-by-side сравнение переводов, персональная полка сохранённых разоблачений.

## Критерии приёмки

- Единая схема `data/exposures/*.json`, старые три источника мигрированы или явно выведены из скоупа.
- Один рендерер (`exposure-case.js`) используется и в каталоге, и в детальном просмотре, и в форме создания.
- Кнопка «+ Новое дело» открывает форму по той же схеме, live-preview идентичен итоговой странице дела.
- Каждое дело открывается по глубокой ссылке, показывает полный набор секций, breadcrumb, TOC, infobox, badges доверия.
- Поиск и фильтры работают, состояние сохраняется в URL.
- Весь markdown, попадающий в DOM, санитизирован.
- Старый рендеринг длинной ленты (`load-researches.js` detail, `renderResearchDetail`, `renderDocumentPage` для эссе) удалён или явно выведен из скоупа.

## Вне скоупа

`js/dashboard.js` / `css/dashboard.css` — не трогать, отдельная незавершённая работа (виджеты `#dashboard`), не связанная с разоблачениями.
