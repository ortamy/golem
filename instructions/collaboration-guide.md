## 👤 КТО МОЖЕТ УЧАСТВОВАТЬ

**НУЛЕВОЙ УРОВЕНЬ — ЧИТАТЕЛИ**
- читают репозиторий
- изучают материалы
- задают вопросы

**ПЕРВЫЙ УРОВЕНЬ — ИССЛЕДОВАТЕЛИ**
- предлагают идеи через `ideas/`
- задают вопросы через `drafts/questions.md`
- сообщают об ошибках

**ВТОРОЙ УРОВЕНЬ — АВТОРЫ**
- пишут исследования
- добавляют термины
- создают черновики

**ТРЕТИЙ УРОВЕНЬ — РЕДАКТОРЫ**
- проверяют файлы через аудит
- исправляют ошибки
- обновляют метаданные

**ЧЕТВЁРТЫЙ УРОВЕНЬ — АДМИНИСТРАТОРЫ**
- управляют репозиторием
- утверждают изменения
- определяют направление

## 📝 КАК ДОБАВИТЬ КОНТЕНТ

### ШАГ 1: ПРОЧИТАЙ

- `README.md` — лицо проекта
- `instructions/RESEARCH-PRINCIPLES.md` — 38 принципов
- `instructions/FORBIDDEN-WORDS.md` — запрещённые слова
- `CONTROL.md` — пульт управления проектом

### ШАГ 2: ВЫБЕРИ ЗАДАЧУ

- открытые задачи в `docs/BACKLOG.md`
- идеи в `ideas/`
- отсутствующие термины в `terminology/`
- недостающие исследования

### ШАГ 3: СОЗДАЙ ЧЕРНОВИК

- создай файл в `drafts/`
- следуй шаблону `instructions/templates/RESEARCH-TEMPLATE.md`
- заполни метаданные

### ШАГ 4: ПРОВЕРЬ СЕБЯ

- заголовок: эмоджи + КАПС
- имя файла: латиница, дефис, КАПСОМ
- нет религионимов
- есть стихи ТаНаХа с переводом

### ШАГ 5: ОТПРАВЬ НА АУДИТ

- напиши Свидетелю (Эд) — используй `instructions/CHAT-AGENT-PROMPT.md`
- или поручи Cline (ИИ-ассистент в VS Code) — используй `instructions/CHAT-ASSISTANT-PROMPT.md`
- дождись проверки по 4 этапам
- исправь замечания

### ШАГ 6: ПУБЛИКУЙ

- перемести файл в нужную папку
- обнови метаданные
- запусти инструменты
- создай pull request

## 💬 КАК ОБЩАТЬСЯ

### В ЧАТЕ С ЭДОМ (СВИДЕТЕЛЕМ)

- используй `instructions/CHAT-AGENT-PROMPT.md` для начала чата
- задавай вопросы прямо
- присылай тексты на проверку

### В CLINE (VS CODE)

- используй `instructions/CHAT-ASSISTANT-PROMPT.md` для начала
- Cline пишет код, правит файлы, запускает скрипты
- для ежедневной рутины: `instructions/CLINE-ROUTINE.md`

### В ISSUES

- сообщай об ошибках
- предлагай улучшения
- задавай вопросы по контенту

### В PULL REQUESTS

- описывай изменения
- ссылайся на связанные issues
- указывай, что проверено

### РАЗРЕШЕНО

- задавать вопросы

## 🔄 ПРОЦЕСС PULL REQUEST

1. форкнуть репозиторий
2. создать ветку `feature/название`
3. добавить файлы
4. запустить инструменты проверки
5. создать pull request
6. дождаться ревью
7. исправить замечания
8. получить апрув
9. слияние

**Пример названия ветки:**
- `feature/add-hesed-terminology`
- `fix/update-metadata`
- `docs/workflow-update`

---

## ✅ ЧЕК-ЛИСТ ПЕРЕД PULL REQUEST

- [ ] файлы в правильных папках
- [ ] имена файлов корректны
- [ ] метаданные заполнены

- [ ] инструменты проверены
- [ ] нет конфликтов с main
