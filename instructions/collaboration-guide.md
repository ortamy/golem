## 🔥 ВВЕДЕНИЕ

Проект «Голем» открыт для всех, кто хочет участвовать в восстановлении истины ТаНаХа.

Этот документ описывает:
- как присоединиться к проекту
- как добавлять контент
- как общаться с командой
- какие правила соблюдать

---

## 📋 ОСНОВНЫЕ ПРИНЦИПЫ

- истина превыше всего
- уважение к участникам
- проверка через ТаНаХ
- отказ от религионимов
- следование процессу

---

## 👤 КТО МОЖЕТ УЧАСТВОВАТЬ

**НУЛЕВОЙ УРОВЕНЬ — ЧИТАТЕЛИ**
- читают репозиторий
- изучают материалы
- задают вопросы

**ПЕРВЫЙ УРОВЕНЬ — ИССЛЕДОВАТЕЛИ**
- предлагают идеи через `ideas/`
- задают вопросы через `drafts/questions.md`
- сообщают об ошибках

**ВТОРОЙ УРОВЕНЬ — АВТОРЫ**
- пишут исследования
- добавляют термины
- создают черновики

**ТРЕТИЙ УРОВЕНЬ — РЕДАКТОРЫ**
- проверяют файлы через аудит
- исправляют ошибки
- обновляют метаданные

**ЧЕТВЁРТЫЙ УРОВЕНЬ — АДМИНИСТРАТОРЫ**
- управляют репозиторием
- утверждают изменения
- определяют направление

---

## 📝 КАК ДОБАВИТЬ КОНТЕНТ

### ШАГ 1: ПРОЧИТАЙ

- `README.md` — лицо проекта
- `instructions/research-principles.md` — 38 принципов
- `instructions/forbidden-words.md` — запрещённые слова
- `instructions/workflow.md` — рабочий процесс

### ШАГ 2: ВЫБЕРИ ЗАДАЧУ

- открытые задачи в `docs/BACKLOG.md`
- идеи в `ideas/`
- отсутствующие термины в `terminology/`
- недостающие исследования

### ШАГ 3: СОЗДАЙ ЧЕРНОВИК

- создай файл в `drafts/`
- следуй шаблону `research-template.md`
- заполни метаданные

### ШАГ 4: ПРОВЕРЬ СЕБЯ

- заголовок: эмоджи + КАПС
- имя файла: латиница, дефис
- нет религионимов
- есть стихи ТаНаХа с переводом

### ШАГ 5: ОТПРАВЬ НА АУДИТ

- напиши Свидетелю (Эд) или редактору
- дождись проверки по 4 этапам
- исправь замечания

### ШАГ 6: ПУБЛИКУЙ

- перемести файл в нужную папку
- обнови метаданные
- запусти инструменты
- создай pull request

---

## 🛠️ КАК ПРОВЕРЯТЬ ЧУЖИЕ ФАЙЛЫ

### АУДИТ ПО 4 ЭТАПАМ

**1. BDIKAH — религионизмы**
- найти запрещённые слова
- предложить замены

**2. MIVDAK — полезность**
- оценить по 23 методам
- сказать, чего не хватает

**3. TIKUN — задачи**
- указать, что нужно исправить
- отметить, что добавить

**4. FACTCHECK — факты**
- проверить утверждения
- указать источники

### ФОРМАТ ОТВЕТА

```markdown
## АУДИТ [НАЗВАНИЕ ФАЙЛА]

### BDIKAH
❌ «Бог» → замена: «Элоhим»

### MIVDAK
➕ добавить корневой разбор

### TIKUN
🔴 исправить ссылку в строке 15

### FACTCHECK
⚠️ дата 325 год — требует подтверждения

### ИТОГ
✅ Готов к публикации после исправлений
```

## 💬 КАК ОБЩАТЬСЯ

### В ЧАТЕ С СВИДЕТЕЛЕМ

- задавай вопросы прямо
- присылай тексты на проверку
- уточняй непонятные моменты

### В ISSUES

- сообщай об ошибках
- предлагай улучшения
- задавай вопросы по контенту

### В PULL REQUESTS

- описывай изменения
- ссылайся на связанные issues
- указывай, что проверено

---

## ⚖️ ПРАВИЛА ПОВЕДЕНИЯ

### РАЗРЕШЕНО

- задавать вопросы
- предлагать идеи
- указывать на ошибки
- спорить, аргументируя ТаНаХом

### ЗАПРЕЩЕНО

- использовать религионимы
- оскорблять участников
- навязывать своё мнение без доказательств
- добавлять контент без аудита
- удалять чужие файлы без обсуждения

---

## 🔄 ПРОЦЕСС PULL REQUEST

1. форкнуть репозиторий
2. создать ветку `feature/название`
3. добавить файлы
4. запустить инструменты проверки
5. создать pull request
6. дождаться ревью
7. исправить замечания
8. получить апрув
9. слияние

**Пример названия ветки:**
- `feature/add-hesed-terminology`
- `fix/update-metadata`
- `docs/workflow-update`

---

## ✅ ЧЕК-ЛИСТ ПЕРЕД PULL REQUEST

- [ ] файлы в правильных папках
- [ ] имена файлов корректны
- [ ] метаданные заполнены
- [ ] нет религионимов
- [ ] инструменты проверены
- [ ] нет конфликтов с main

## 📞 КОНТАКТЫ
