## 🔥 ВВЕДЕНИЕ

Этот документ описывает стандартный рабочий процесс для добавления, проверки и публикации файлов в репозитории «Голем».

Все участники следуют этому процессу для обеспечения качества и согласованности.

---

## 📋 ОСНОВНЫЕ ЭТАПЫ

### ЭТАП 1: ПЛАНИРОВАНИЕ

- открыть `BACKLOG.md` — посмотреть текущие задачи
- открыть `ideas/` — поискать готовые концепции
- открыть `TODO.md` (если есть) — текущие приоритеты

**Результат:** понятная цель и понимание, не дублируется ли работа

---

### ЭТАП 2: СОЗДАНИЕ ЧЕРНОВИКА

- создать файл в `drafts/` с рабочим названием
- заполнить базовую структуру (заголовок, метаданные)
- написать черновик исследования

**Правила черновика:**
- не требует идеального оформления
- можно использовать временные формулировки
- метаданные минимальны (хотя бы тема)

**Результат:** черновик готов к проверке

---

### ЭТАП 3: САМОПРОВЕРКА

Перед отправкой на аудит автор проверяет:

- есть ли заголовок с эмоджи и КАПСом
- есть ли блок метаданных (Файл, Версия, Дата, Статус, Тема)
- нет ли религионимов в авторской речи
- имя файла соответствует правилам (латиница, дефис)

**Результат:** черновик соответствует минимальным стандартам

---

### ЭТАП 4: АУДИТ (4 ЭТАПА)

Аудит проводит Свидетель (Эд) или другой проверяющий.

**4.1 BDIKAH — проверка на религионизмы**

- сканирование текста на запрещённые слова
- выдача замен
- маркеры: ❌ нарушение, ⚠️ под вопросом, ✅ допустимо

**4.2 MIVDAK — аудит полезности**

- оценка по 23 методам
- что можно добавить, что убрать
- маркеры: ➕ добавить, 🟡 обновить, 🔴 не подходит

**4.3 TIKUN — задачи по обновлению**

- что исправить в существующих файлах
- что создать нового
- маркеры: ✅ выполнено, 🔴 не выполнено, ⏳ в процессе

**4.4 FACTCHECK — проверка фактов**

- проверка утверждений
- маркеры: ✅ подтверждено, ⚠️ гипотеза, ❌ опровергнуто

**Результат:** файл одобрен или отправлен на доработку

---

### ЭТАП 5: ДОРАБОТКА (ЕСЛИ НУЖНО)

- исправить замечания из аудита
- дополнить недостающие разделы
- проверить ссылки и цитаты

**Результат:** исправленная версия

---

### ЭТАП 6: ПУБЛИКАЦИЯ

- переместить файл из `drafts/` в нужную папку:
  - `terminology/` — для разбора слов
  - `researches/` — для исследований явлений
  - `instructions/` — для новых инструкций
  - `tools/` — для новых инструментов
- обновить метаданные (дату, версию, статус)
- запустить `python3 tools/sync-structure.py`
- запустить `python3 tools/generate-glossary.py` (если термин)
- запустить `python3 tools/generate-nav.py`

**Результат:** файл опубликован

---

### ЭТАП 7: КОММИТ И ПУШ

```bash
git add .
git commit -m "Добавлено: [название файла]"
git push
```

**Результат:** изменения в репозитории

## 🛠️ ИНСТРУМЕНТЫ ДЛЯ ПРОВЕРКИ

**ПЕРЕД ПУБЛИКАЦИЕЙ ЗАПУСТИТЬ:**

```bash
cd tools

python3 check-naming.py        # проверить имена файлов
python3 validate-metadata.py   # проверить метаданные
python3 check-links.py         # проверить ссылки
python3 find-duplicates.py     # проверить дубликаты
```

**ПОСЛЕ ПУБЛИКАЦИИ ЗАПУСТИТЬ:**

```bash
python3 sync-structure.py      # обновить structure.md
python3 generate-glossary.py   # обновить глоссарий
python3 generate-nav.py        # обновить навигацию
python3 stats-report.py        # обновить статистику
```

## 🚫 ЧЕГО НЕ ДЕЛАТЬ

- не создавать файлы с русскими буквами в имени
- не использовать таблицы вместо списков
- не писать религионизмы в авторской речи
- не создавать дубликаты существующих исследований
- не коммитить без проверки инструментами
- не удалять файлы без обновления structure.md

## 🔄 ВЕРСИОНИРОВАНИЕ

**Формат версии:** `major.minor`

- **major** — большие изменения, новая структура, пересмотр концепций
- **minor** — добавление файлов, исправления, обновления

**Когда обновлять:**
- новый файл — minor +1
- исправление ошибок — minor +1
- массовые правки — minor +1
- пересмотр принципов — major +1
